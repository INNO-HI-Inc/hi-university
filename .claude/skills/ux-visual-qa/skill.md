# UX Visual QA

구현 결과물을 참고 디자인과 비교하여 차이점을 정량적으로 지적하는 스킬.

## 워크플로우

### Step 1: 스크린샷 수집

1. **참고 스크린샷** 열기 (Read로 이미지 파일)
2. **구현 스크린샷** 캡처 (Playwright MCP 또는 직접 URL 확인)

### Step 2: 비교 분석

아래 체크리스트를 **반드시 전부** 확인한다:

#### A. 레이아웃 구조 (P0)
- [ ] 전체 그리드 컬럼 수 일치
- [ ] 히어로 위치 (풀 width vs 컬럼 안)
- [ ] 사이드바 유무 및 위치
- [ ] 2단 레이아웃 시작점 (히어로부터? 카드부터?)

#### B. 크기/비율 (P1)
- [ ] 좌/우 컬럼 비율
- [ ] 히어로 높이
- [ ] 카드 크기 (너비 x 높이)
- [ ] 사이드바 폭
- [ ] 버튼 크기

#### C. 여백/간격 (P1)
- [ ] 섹션 간 gap
- [ ] 카드 간 gap
- [ ] 내부 padding
- [ ] 네비바 높이

#### D. 타이포그래피 (P2)
- [ ] 헤딩 크기/굵기
- [ ] 본문 크기/컬러
- [ ] 줄간격

#### E. 컬러/시각 (P2)
- [ ] 배경색
- [ ] 보더
- [ ] 그림자
- [ ] border-radius

#### F. 콘텐츠 (P2)
- [ ] 텍스트 내용
- [ ] 일러스트/아이콘
- [ ] 데이터 표시

#### G. 인터랙션 (P3)
- [ ] 호버 효과
- [ ] 애니메이션
- [ ] 전환 효과

### Step 3: 피드백 작성

```markdown
# QA 피드백 #{n}

## P0 (구조 오류)
- [영역] 문제
  - 참고: ...
  - 현재: ...
  - 수정: 구체적 Tailwind 클래스 제안

## P1 (크기 오류)
...

## P2 (디테일)
...

## P3 (폴리시)
...

## 판정: PASS / FAIL (P0/P1 이슈 유무 기준)
```

### Step 4: 결과 저장

피드백을 `_workspace/qa_feedback_{n}.md`에 저장한다.

## 판정 기준

- **FAIL**: P0 또는 P1 이슈가 1개 이상 → 디자이너 수정 필요
- **PASS**: P0/P1 없음, P2 3개 이하 → 완료
- **CONDITIONAL PASS**: P0/P1 없음, P2 4개 이상 → 디자이너 판단
